# Ревью архитектуры и спецификаций (Q1)

Ниже представлена таблица с выявленными архитектурными рисками, уязвимостями и расхождениями в документации, а также предложенными решениями.

| №   | Локация (Файл / Блок)                                | Проблема / Риск                                                                                                     | Решение / Действие                                                                                                           |
| --- | ---------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------- |
| 1   | `step-2/README.md` → `users.ts`                      | Циклический импорт `users.ts` ↔ `audit.ts`. Во время выполнения `relations` могут получить `undefined`.             | Разорвать циклическую зависимость (например, вынести общие типы/переменные).                                                 |
| 2   | `step-2/README.md` → `profiles.countryId`            | `countryId` — `varchar(36)`, а в `businesses` — `integer`. Невозможно сделать JOIN.                                 | Сразу объявить `countryId` как `integer`.                                                                                    |
| 3   | `step-2/README.md` → `memberships.uxUserActive`      | Индекс по `(userId, type, status)` ломает историю подписок при обновлении (потеря истории или двусмысленность).     | Сделать отдельную таблицу для истории ИЛИ уникальный индекс только по `(userId, type)`.                                      |
| 4   | `B05/04-webhook-endpoint-and-idempotency.md`         | Свойство `s.current_period_end` удалено в новых версиях Stripe API. Вызовет ошибку `TypeError`.                     | Зафиксировать `apiVersion` в конфиге Stripe. Доставать поле через `items` (`subscription.items.data[*].current_period_end`). |
| 5   | `B04/03-roles-and-profile-attributes.md`             | Вызов `auth()` из `@clerk/nextjs/server` в Clerk v6 стал асинхронным. Код упадет.                                   | Использовать асинхронный вызов `await auth()`.                                                                               |
| 6   | `B10/03-verify-card-public-route.md`                 | `revalidate = 120` опасен: заблокированная карта может 2 минуты показываться как `ACTIVE` (уязвимость).             | Использовать `cache: 'no-store'` или `revalidateTag('card:<number>')` + инвалидация при обновлении админом.                  |
| 7   | `B10/03` (Верификация карты)                         | Нет rate-limit и Turnstile на публичном роуте с предсказуемыми номерами. Возможен перебор (enumeration).            | Добавить Turnstile, Upstash rate-limit (по IP+number), убедиться в не-детерминированности номеров (entropy ≥ 64 bit).        |
| 8   | `B15/01-next-intl-skeleton`                          | Задан только `Locales = ['en']`, хотя в ТЗ RU / EN / UK. Позже внедрить локали будет сложно.                        | Зафиксировать все локали до внедрения маршрутизации (шаг B06), так как `/[locale]/...` влияет на весь App Router.            |
| 9   | `B12/05-introductions-management-stub`               | Статусы (SUBMITTED, APPROVED...) переключаются без контроля переходов (можно из CLOSED в DRAFT).                    | Внедрить State Machine (машину состояний) для статусов и аудит каждого перехода.                                             |
| 10  | `step-3/04-business-crud/01-create-business-form.md` | Пользователь может выставить `visibility: PUBLIC` для офферов (по ТЗ доступно только админам).                      | Запретить пользователям делать офферы публичными в форме создания (оставить только `PRIVATE_AFTER_LOGIN`).                   |
| 11  | `step-1` vs `step-2` vs `step-3`                     | Дублирование и конфликты схем в разных шагах (например, B03, step-2, step-3/02, B05). Агент может запутаться.       | Создать `INDEX.md`, четко описывающий приоритеты (`step-2` заменяет B03, `step-3` заменяет `B04/B05`).                       |
| 12  | Все промпты                                          | Нет блоков `Inputs` (предварительные условия) и `Outputs` (результаты, кроме файлов). Страдает идемпотентность.     | Добавить четкие блоки `Inputs` и `Outputs` во все промпты.                                                                   |
| 13  | Все промпты                                          | Нет фиксации версий зависимостей (Next 15.x, React 19.x, Clerk ^6 и т.д.). Риск поломки в будущем из-за обновлений. | Явно указать версии всех критичных библиотек во избежание проблем при сборке.                                                |
| 14  | Все промпты                                          | Нет блока `Rollback / Undo` (инструкций по откату в случае ошибки агента).                                          | Добавить шаги по откату изменений на случай падения генерации/скрипта.                                                       |
| 15  | `docs/CONTEXT.md`                                    | Заголовок `Context-ID: kclub-mvp-<commit-sha>` пуст.                                                                | Заполнить `commit-sha` актуальным значением.                                                                                 |
| 16  | `docs/SPEC.md` vs реальные промпты                   | В ТЗ нет явных требований к темам, а промпты делают dark/gold тему (без светлой).                                   | Согласовать требования к темам между SPEC и промптами.                                                                       |
